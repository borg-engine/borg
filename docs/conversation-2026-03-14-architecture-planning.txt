## What we do today

The MCP system has two halves:

**Server side** (what we built, works well): `sidecar/borg-mcp/server.js` — a proper MCP server exposing 12 tools (search_documents, read_document, check_coverage, create_task, upload_to_knowledge, etc.) over stdio. This is solid and domain-specific. No complaints here.

**Client side** (what we don't control): We never speak MCP protocol ourselves. The flow is:

1. `claude.rs` builds a JSON blob describing our MCP servers (command, args, env vars)
2. Writes it to `.mcp.json` in the session dir
3. Passes `--mcp-config .mcp.json` to the Claude Code CLI subprocess
4. Claude Code spawns the MCP server processes internally and talks MCP protocol to them
5. We see nothing — Claude decides when to call tools, handles errors internally

Same pattern in `codex.rs` — we push `mcp_servers.borg.command`, `mcp_servers.borg.args` etc. as config args, and Codex handles it.

## Why it's bad

**We have zero visibility into what happens.** Once Claude/Codex spawns our MCP server:

- **No timeout control** — if `search_documents` hangs for 60 seconds because Vespa is slow, we don't know. Claude Code has its own internal timeouts but we can't configure them per-tool.
- **No error observability** — if the MCP server crashes or returns errors, those show up in Claude's reasoning (maybe) but never in our logs. We can't alert on repeated failures.
- **No metrics** — we can't track which MCP tools are being called, how often, latency distribution, error rates. We're flying blind on tool usage.
- **No rate limiting** — can't prevent a runaway agent from hammering `search_documents` 200 times in a loop.
- **No tool-level access control** — once MCP is wired, the agent can call any tool. We filter via `--allowed-tools` at the Claude level, but that's a string allowlist, not programmatic control.
- **Lifecycle coupling** — the MCP server lives and dies with the Claude subprocess. If we want to share a warm MCP connection across multiple agent invocations (e.g. resumed sessions), we can't — each spawn is fresh.

## Proposed solution

An **in-process MCP client** in Rust that sits between our agent backends and the MCP servers:

```
Before:  Borg → spawns Claude CLI → Claude spawns MCP server → tools
After:   Borg → spawns MCP servers (long-lived) → agent calls go through our proxy → tools
```

Concretely:
- Borg starts MCP servers on its own (stdio or HTTP transport) and holds the connections
- When an agent needs tools, we provide them as native tool definitions (not via `--mcp-config`)
- Tool calls from the agent come back to us, we forward to the MCP server, apply timeouts/logging/rate-limits, return results
- MCP servers stay warm across session resumes — no cold-start per agent invocation
- Full observability: log every call, measure latency, count errors, emit metrics

**But** — this is a significant architecture change. It means intercepting all tool calls from the agent, which changes how we integrate with Claude Code (we'd need to use the API or agent SDK rather than CLI subprocess, or use a proxy pattern). That's why I put it last on the list — highest value long-term but highest effort.

The pragmatic middle ground would be: keep the current architecture but add a **sidecar logging proxy** that wraps our MCP servers, captures all calls/responses/latency, and forwards to our observability stack. Same architecture, just instrumented.

- Rust → spawns `claude` CLI → CLI spawns MCP bun servers → raw NDJSON stdout → manual parsing
- No observability into tool calls
- No retry/fallback
- Session IDs extracted via regex from NDJSON
- .mcp.json + CLAUDE.md written to disk per invocation
- Separate code paths for chat vs pipeline

**After (Agent SDK):**
- Rust → spawns agent bridge (bun, long-lived) → Agent SDK handles everything
- Hooks give us every tool call, timing, errors
- Structured events instead of NDJSON parsing
- Session management built in
- MCP servers passed as config (no file writing)
- Same path for chat and pipeline, different options

## Full task list

### Phase 1: Agent Bridge foundation
1. **`sidecar/agent-bridge/`** — thin TypeScript wrapper around `@anthropic-ai/claude-agent-sdk`
   - Long-lived bun process, communicates with Rust via JSON over stdio
   - Accepts query requests (prompt, options, hooks config)
   - Streams structured events back (tool calls, text deltas, results, usage)
   - Passes through MCP servers config (borg-mcp, lawborg-mcp unchanged)

2. **`borg-agent/src/bridge.rs`** — Rust client for the agent bridge
   - Spawns and manages the bridge process lifecycle
   - Sends queries, receives typed events via serde
   - Replaces both `ClaudeBackend` and chat's `Command::new("claude")`

3. **Provider abstraction** — env-var driven, trait-gated
   - `ProviderConfig` enum: `Bedrock { region, ... }`, `Vertex { project_id, ... }`, `Direct { api_key }`
   - Bridge passes through env vars (`CLAUDE_CODE_USE_BEDROCK=1`, `AWS_REGION`, etc.)
   - Bedrock first, Vertex stub for later
   - Config: `BORG_PROVIDER=bedrock|vertex|direct`

### Phase 2: Migrate chat agents
4. **Replace `routes/chat.rs` subprocess spawning**
   - Remove `Command::new("claude")` + args building (~200 lines)
   - Remove .mcp.json file writing
   - Remove NDJSON parsing + session ID regex extraction
   - Use bridge with `allowedTools`, `mcpServers`, `systemPrompt`, `maxTurns`
   - Hook events → SSE chat stream (replaces raw NDJSON forwarding)
   - Session resumption via SDK's `resume` option

### Phase 3: Migrate pipeline agents
5. **Replace `claude.rs` CLI backend**
   - Remove `Command::new(claude_bin)` + bwrap wrapping
   - Bridge query with `cwd` set to worktree path
   - Phase-specific `allowedTools` and `disallowedTools`
   - `permissionMode: "bypassPermissions"` for pipeline (already sandboxed)
   - MCP servers for BorgSearch access
   - Session continuity across phases via SDK resume

6. **Codex backend** — evaluate whether to keep or migrate
   - If Agent SDK supports OpenAI models via env vars, migrate
   - If not, keep `codex.rs` as fallback for OpenAI-backed tasks

### Phase 4: Observability via hooks
7. **Hook handler implementation**
   - `PreToolUse` → log tool name, input, timestamp; rate limiting check
   - `PostToolUse` → log result, latency, success/error; emit metrics
   - `Stop` → log stop reason, total token usage, cost estimate
   - Wire into existing tracing spans + stream manager
   - Dashboard: show tool call timeline per chat/task (new UI later)

### Phase 5: ReliableProvider
8. **Retry/backoff wrapper** around bridge invocations
   - Error classification (adapted from ZeroClaw's logic):
     - Retryable: 429 rate limit, 5xx server errors, timeouts
     - Non-retryable: 400 bad request, 401 auth, context overflow
   - Exponential backoff with jitter, Retry-After header respect
   - For pipeline: smart phase retry (don't restart from scratch)
   - For chat: transparent retry, user sees brief delay not error

### Phase 6: Cron/scheduling
9. **`borg-core/src/cron.rs`** — persistent job scheduler
   - PostgreSQL-backed job store (table: `cron_jobs`, `cron_runs`)
   - Polling loop with configurable interval, missed-tick skipping
   - Job types: `AgentTask` (create pipeline task), `Shell` (run command)
   - Per-job retry with backoff + jitter
   - Delivery: notify chat thread on completion
   - API: CRUD endpoints + dashboard UI
   - Use cases: nightly re-index, weekly compliance scan, scheduled reports

### Phase 7: Secret encryption
10. **ChaCha20-Poly1305 for secrets at rest**
    - Encrypt API keys in DB (linked credentials, api_keys table)
    - Key file at `$DATA_DIR/.secret_key` (0600 perms)
    - Migration: encrypt existing plaintext keys on startup
    - Leak detector: warn if unencrypted secrets found in config/env

### Phase 8: Cleanup
11. **Remove old code**
    - Delete `claude.rs` CLI subprocess backend
    - Delete NDJSON event parser (`borg-agent/src/event.rs`?)
    - Delete .mcp.json generation from `mcp.rs` (keep `build_mcp_servers_json` adapted for bridge format)
    - Delete manual session ID extraction
    - Update `codex.rs` or delete if Agent SDK covers it
    - Update deploy scripts (agent-bridge needs bun + SDK installed)

## Answers to your questions

### Claude subscription auth
Yes, the Agent SDK supports Claude subscription auth (OAuth tokens from Claude Pro/Max/Teams/Enterprise). It's just Claude Code under the hood — same auth flows. We'd support three auth modes:
- **Claude subscription** (OAuth token) — current default, still works
- **Bedrock** (AWS IAM credentials) — for ZDR
- **Vertex** (GCP credentials) — future addition

No auth flow changes needed. The Agent SDK inherits whatever `CLAUDE_CODE_OAUTH_TOKEN` or `CLAUDE_CODE_USE_BEDROCK` env vars are set.

### Scheduling vs auto tasks
You're right — we already have the pipeline task system with auto-enqueue. What we don't have is *time-triggered* tasks (e.g., "re-index this corpus every night at 2am", "run a compliance check every Monday"). Is that something you actually need, or is the current manual/chat-triggered pipeline sufficient? If the current flow covers it, I'll drop cron from the list.

### OpenAI / Codex support
The Codex CLI harness being open source Rust is interesting. Three options:

**A) Agent SDK only** — The Agent SDK might support OpenAI models if Claude Code does (via `--model` flag). Need to verify.

**B) Keep Codex backend** — Our current `codex.rs` spawns the Codex CLI. It works. We could keep it alongside the Agent SDK bridge for when users want OpenAI models.

**C) Integrate Codex harness** — Since it's Rust and open source, we could integrate it as a native backend. No TypeScript bridge needed for OpenAI. This is the most powerful option but biggest lift.

My instinct is B for now (keep what works), evaluate C later if we want deeper OpenAI integration. What's your thinking?

### "New UI later" for tool call timeline
I meant we'd add a panel in the dashboard that shows the sequence of tool calls an agent made during a chat or task — like a timeline showing "searched documents (340ms) → read file #42 (120ms) → created task (80ms)". The hook data makes this possible. "Later" just meant it's a dashboard feature we'd build after the backend work. Not a separate project — just sequenced after the core migration. Should I include it in the task list as a concrete item?

### Keep existing architecture as an option?
This is the most important question. I think yes — make it modular. Here's why:

The Agent SDK is great for Claude. But the container-based architecture (`container/Dockerfile` + `entrypoint.sh`) is **provider-agnostic** — it can run any CLI agent in a sandbox. If tomorrow there's a new agent runtime (Gemini agent, open-source alternative, custom fine-tuned model with its own harness), the container approach just works.

The clean design would be:

```
AgentBackend (trait)
├── AgentSdkBackend    — Claude via Agent SDK (primary)
├── CodexBackend       — OpenAI via Codex CLI (existing)
├── ContainerBackend   — any CLI agent in Docker/bwrap (existing, generalized)
└── OllamaBackend      — local models (existing)
```

The Agent SDK becomes the **primary** path for Claude, but the container backend stays as a general-purpose fallback. This means we're not ripping out old code — we're adding a better option alongside it and defaulting to it. The task list gets simpler too: no "delete old code" phase, just "add new backend + make it default."

This modular approach is also cleaner from a config perspective — users/deployments choose their backend per-task or globally:
```
BORG_BACKEND=agent-sdk          # default, Claude via Agent SDK
BORG_BACKEND=codex              # OpenAI via Codex CLI
BORG_BACKEND=container          # generic, any agent in Docker
BORG_BACKEND=ollama             # local models
```
